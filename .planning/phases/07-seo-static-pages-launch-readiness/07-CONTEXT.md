# Phase 7: SEO, Static Pages & Launch Readiness — Context

**Gathered:** 2026-05-16
**Status:** Ready for planning

<domain>
## Phase Boundary

Phase 7 delivers: (1) redesigned static/legal pages aligned to the site design system, (2) full SEO infrastructure — per-page editable meta stored in DB, sitemap.xml auto-generated, robots.txt configured, JSON-LD schema markup in page source, (3) Calendly booking popup wired to all CTAs site-wide, and (4) page speed audit before launch.

This phase does NOT add new content sections, new page types, or new admin features beyond the SEO editor and Calendly URL setting.

</domain>

<decisions>
## Implementation Decisions

### Calendly Integration (SEO-05)
- **D-01:** ALL CTAs site-wide trigger the Calendly booking popup — not selective. This includes: homepage hero CTA, all service page CTAs, Connect CTA block (used at bottom of most pages), navbar "Book a Call" button, and any inline CTAs on blog/guide/testimonials pages that use the standard CTA text.
- **D-02:** Add a new `calendly_url` settings key (separate from `default_cta_link`). When `calendly_url` is set in admin Settings, Calendly popup fires. When empty (no URL configured), CTAs fall back to `default_cta_link` so there is zero regression on a fresh deployment.
- **D-03:** `connect-cta.tsx` already fetches `default_cta_link` from `settings.php` — extend this fetch to also read `calendly_url`. When populated, render `<CalendlyButton>` instead of `<Link>`.
- **D-04:** `CalendlyButton` is a new client component (`components/ui/calendly-button.tsx`) wrapping `react-calendly`'s `PopupWidget`. It must be loaded via `dynamic(() => import(...), { ssr: false })` to avoid static build failures.

### Static Pages (STATIC-01..05)
- **D-05:** **Brand voice:** Quirky/creative copy for user-facing static pages (404, thank-you). `not-found.tsx` can use something like "Lost in the automation void?" with helpful navigation links. The existing thank-you page "Transmission Success" tone is acceptable — the redesign updates the layout to match the design system, not necessarily the copy voice.
- **D-06:** **Legal pages:** Privacy Policy and Terms & Conditions get professionally written content appropriate for a digital marketing automation agency — covering AI data processing, third-party tool usage (Google Analytics, Calendly), GDPR-adjacent data handling, IP ownership, SLAs, and liability limits.
- **D-07:** `app/not-found.tsx` at the app root (NOT `/404/page.tsx`) compiles to `out/404.html` in static export — confirmed approach. It renders within root layout so Navbar/Footer are inherited automatically; do NOT import them manually.
- **D-08:** `app/error.tsx` is a GlobalError component — redesign to match design system without breaking the GlobalError interface (`reset` function prop).

### SEO Meta Admin (SEO-01)
- **D-09:** New `seo_meta` DB table with `page_key VARCHAR(100) PRIMARY KEY`, `seo_title VARCHAR(255)`, `meta_description TEXT`, `og_image TEXT`.
- **D-10:** New `backend/api/seo_meta.php` endpoint: GET by `?page={key}`, POST `action=save_seo_meta` with page_key + fields.
- **D-11:** New admin page at `/admin/seo` — dropdown selector for common pages (hardcoded list: home, blog, case-studies, guides, services, testimonials, contact-us, privacy-policy, terms-and-conditions, plus service slugs and blog slugs) + title/description/og-image fields.
- **D-12:** OG image input is a **URL text field** — admin pastes an image URL (no file upload). Simple, no new upload endpoint, works with static export.
- **D-13:** Changes to SEO meta require a site rebuild to appear in page source (build-time constraint). Admin SEO page must display a clear notice: *"Changes take effect after the next site rebuild."*

### SEO Infrastructure (SEO-02/03/04)
- **D-14:** `next-sitemap` (v4.2.3) as a `postbuild` npm script — outputs `sitemap.xml` and `robots.txt` to `out/`. Native `sitemap.ts` / `robots.ts` skipped (known bug with `output: 'export'` in Next.js 15/16).
- **D-15:** robots.txt disallows `/admin` and `/admin/*`; allows everything else. Generated by `next-sitemap` config with `generateRobotsTxt: true`.
- **D-16:** JSON-LD schema via native `<script type="application/ld+json">` in Server Components only (NOT `next/script`). XSS prevention: `.replace(/</g, '\\u003c')` on the stringified JSON.
- **D-17:** Organization schema in `app/layout.tsx` (sitewide). Service schema in `app/services/[slug]/page.tsx` server wrapper. Review/testimonials aggregate schema on the testimonials page server wrapper if applicable.

### generateMetadata Pattern (SEO-01)
- **D-18:** All public page `page.tsx` files that are currently `"use client"` need a thin server wrapper that: (a) exports `generateMetadata` fetching from `seo_meta.php`, and (b) renders the existing client component. For pages already structured as server wrappers (e.g., `blog/[slug]/page.tsx`), `generateMetadata` is added directly.
- **D-19:** Fallback metadata strings are defined per page (e.g., `'Blog — Digi Pexel'` for the blog listing page) so a page never has a blank `<title>` if the API is down.

### Claude's Discretion
- Schema markup depth per service page (how many fields to include in Service schema) — Claude decides based on schema.org standard fields and what's available in the DB.
- Exact copy for 404 and thank-you page redesigns — Claude decides, keeping the creative/quirky brand voice.
- Which blog/service/case-study slugs to include in the admin SEO dropdown vs. asking admin to type manually — Claude decides based on practicality.
- Page speed optimizations for SEO-06 — Claude audits and applies what's addressable in code (image hints, font preload, etc.); manual Lighthouse audit documents the outcome.

</decisions>

## Existing Code Insights

- `safeFetch()` from `@/lib/utils` — use for all runtime API calls
- `API_BASE_URL` from `@/lib/constants` — use in `generateMetadata` for build-time fetch too

### Established Patterns
- Settings keys use snake_case string storage in the `settings` table (e.g., `'whatsapp_enabled'`, `'default_cta_link'`) — `calendly_url` follows the same pattern
- Admin pages all use `AdminLayout` wrapper — new `/admin/seo` page must too
- Server wrapper + client component split (used in `blog/[slug]/page.tsx`) — the pattern to follow when adding `generateMetadata` to currently-client-only pages
- `INSERT IGNORE` in init_db.php for all seed rows

### Integration Points
- `app/layout.tsx` — add Organization JSON-LD script tag here
- `frontend/package.json` — add `postbuild` script: `"postbuild": "next-sitemap"`
- `backend/init_db.php` — add `seo_meta` table + `calendly_url` seed in settings
- `admin` sidebar (`admin-layout.tsx`) — add SEO entry with `Search` icon at `/admin/seo`
