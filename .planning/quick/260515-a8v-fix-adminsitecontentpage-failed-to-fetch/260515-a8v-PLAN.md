# Quick Task 260515-a8v: Fix Failed to Fetch Dev Overlay Error

## Root Cause

The Chrome extension `frame_ant.js` monkey-patches `window.fetch` and attaches a `.then()` handler to the fetch promise without a corresponding `.catch()`. When the backend API is unreachable, `fetch` rejects with "Failed to fetch". Our `safeFetch` try-catch handles this correctly and returns `{ status: "error" }` — but the extension's `.then()` chain (with no `.catch()`) triggers a browser `unhandledrejection` event before our handler runs. Next.js dev overlay listens for global `unhandledrejection` events and shows the red error screen.

## Fix

In `safeFetch` (`frontend/src/lib/utils.ts`), immediately attach a no-op `.catch(() => {})` to the fetch promise synchronously before `await`-ing it. This marks the rejection as handled from the browser's perspective so `unhandledrejection` never fires, regardless of what the extension does with the same promise.

## Tasks

### Task 1: Suppress premature unhandledrejection in safeFetch

**File:** `frontend/src/lib/utils.ts`

**Change:** Replace `const res = await fetch(url, {...})` with:
```js
const fetchPromise = fetch(url, { ...options, signal: controller.signal });
fetchPromise.catch(() => {}); // suppress unhandledrejection from extension monkey-patches
const res = await fetchPromise;
```

**Verify:** The try-catch still handles all errors from `await fetchPromise`. The no-op `.catch()` only prevents the premature `unhandledrejection` browser event — it does not swallow errors for our code.

**Done:** `utils.ts` modified with the three-line pattern replacing the single `await fetch(...)` line.
