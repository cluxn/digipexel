<objective>
Fix intermittent "Jest worker encountered 2 child process exceptions, exceeding retry limit" crashes.

Purpose: Jest spawns one worker process per CPU core by default. On this Windows machine, each worker loads the full Next.js SWC transform + React 19 + jsdom into its own V8 heap. The service-page test alone consumes ~120 MB per worker; with 4+ parallel workers the aggregate OOM pressure kills individual workers mid-run. The fix caps concurrent workers, sets a per-worker memory limit, adds --forceExit (the async ServicePage component leaves open handles), and wires a proper `npm test` script.

Output: Stable test run — `npm test` from `frontend/` passes all 20 tests every time.
</objective>

<task type="auto">
  <name>Task 1: Add npm test script with worker constraints</name>
  <files>frontend/package.json</files>
  <action>
    Add a `"test"` script to the `"scripts"` block in `frontend/package.json`:

    ```json
    "test": "jest --maxWorkers=2 --forceExit"
    ```

    Rationale:
    - `--maxWorkers=2` caps parallel V8 heaps. With service-page at ~120 MB per worker and heap growing to 168 MB by the 5th suite, 4+ workers exceed available RAM and trigger the crash.
    - `--forceExit` prevents the runner from hanging after the async `ServicePage` component (which is an async Server Component that React logs warnings about in JSDOM) leaves unresolved microtasks open.
    - Do NOT add `--runInBand` — that serializes all suites into one process, which causes the heap to climb to 168 MB+ in a single process and trades worker crashes for single-process OOM on CI.
    - Do NOT add `--coverage` to the default test script — coverage with v8 provider significantly increases memory per worker.

    The existing `"scripts"` block already has `"dev"`, `"build"`, `"start"`, `"lint"`. Add `"test"` after `"lint"`.
  </action>
  <verify>
    <automated>cd frontend && node -e "const s = require('./package.json').scripts; if (!s.test) throw new Error('test script missing'); if (!s.test.includes('--maxWorkers=2')) throw new Error('--maxWorkers=2 missing'); if (!s.test.includes('--forceExit')) throw new Error('--forceExit missing'); console.log('OK:', s.test);"</automated>
  </verify>
  <done>package.json has `"test": "jest --maxWorkers=2 --forceExit"` in scripts</done>
</task>

<task type="auto">
  <name>Task 2: Add workerIdleMemoryLimit to jest.config.js</name>
  <files>frontend/jest.config.js</files>
  <action>
    Add `workerIdleMemoryLimit: "256MB"` to the config object passed to `createJestConfig(...)` in `frontend/jest.config.js`.

    Current content:
    ```js
    const nextJest = require("next/jest");
    const createJestConfig = nextJest({ dir: "./" });
    module.exports = createJestConfig({
      coverageProvider: "v8",
      testEnvironment: "jsdom",
      moduleNameMapper: {
        "^@/(.*)$": "<rootDir>/src/$1",
      },
      setupFilesAfterEnv: ["<rootDir>/jest.setup.ts"],
    });
    ```

    Updated content (add `workerIdleMemoryLimit` as first key for visibility):
    ```js
    const nextJest = require("next/jest");
    const createJestConfig = nextJest({ dir: "./" });
    module.exports = createJestConfig({
      workerIdleMemoryLimit: "256MB",
      coverageProvider: "v8",
      testEnvironment: "jsdom",
      moduleNameMapper: {
        "^@/(.*)$": "<rootDir>/src/$1",
      },
      setupFilesAfterEnv: ["<rootDir>/jest.setup.ts"],
    });
    ```

    `workerIdleMemoryLimit` tells Jest to recycle a worker process when its heap exceeds 256 MB between tests — before it reaches OS-level OOM and crashes with the "child process exceptions" error. The value 256 MB is comfortably above the observed 168 MB peak while low enough to trigger recycling before catastrophic failure.

    Do NOT add `--runInBand` here or set `maxWorkers` in the config file — `maxWorkers` is already set in the npm script and config-file values have lower precedence than CLI flags; duplication causes confusion.
  </action>
  <verify>
    <automated>cd frontend && npx jest --no-coverage --forceExit 2>&1 | tail -6</automated>
  </verify>
  <done>All 5 test suites pass (20 tests), no "child process exceptions" error, run completes cleanly</done>
