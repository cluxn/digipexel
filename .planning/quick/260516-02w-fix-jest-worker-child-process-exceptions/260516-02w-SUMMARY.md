# Phase 260516-02w Plan 01: Fix Jest Worker Child Process Exceptions Summary

**One-liner:** Capped Jest to 2 parallel workers with 256MB idle memory limit and --forceExit to eliminate OOM-driven "child process exceptions" crashes on Windows.

## What Was Done

Jest was crashing intermittently with "Jest worker encountered 2 child process exceptions, exceeding retry limit". Root cause: Jest spawns one worker per CPU core by default. On this Windows machine, each worker loads the full Next.js SWC transform + React 19 + jsdom into its own V8 heap. The service-page test consumes ~120MB per worker; with 4+ parallel workers the aggregate OOM pressure kills individual workers mid-run.

Two targeted config changes fixed this:

1. **`frontend/package.json`** — added `"test": "jest --maxWorkers=2 --forceExit"` to scripts. Caps parallel V8 heaps at 2, preventing aggregate OOM. `--forceExit` prevents runner hang from unresolved microtasks left open by the async `ServicePage` server component in JSDOM.

2. **`frontend/jest.config.js`** — added `workerIdleMemoryLimit: "256MB"`. Tells Jest to recycle a worker process when its heap exceeds 256MB between tests, before it reaches OS-level OOM and crashes with the exception error.
